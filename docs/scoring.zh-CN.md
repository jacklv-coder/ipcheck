# 评分与结论规则

[中文 README](../README.zh-CN.md) · [English](scoring.md)

`ipcheck` 同时给出两类信号：

- 已测服务结论：`GOOD`、`FAIR`、`POOR` 或 `BLOCKED`；provider 专用路径无法
  安全自动探测时为 `SKIPPED`；
- 使用 `rule_v4` 计算的 0–100 开发适配分。

分数是透明的启发式规则，不代表用户百分位，也不评价模型智能水平。

TTFB 来自无凭据协议探测，包含 DNS、代理、TLS、网络与网关入口耗时，但不
包含认证和模型生成。P95 使用最近秩算法；默认三次采样时，它实际上等于最慢
样本。抖动是各样本相对中位数的均方根偏差。比较接近的结果时建议增加采样数。

## 服务结论

| 结果 | 默认规则 |
| --- | --- |
| `GOOD` | 主路径 100% 成功，TTFB 中位数低于 800 ms，抖动低于 1,000 ms |
| `FAIR` | 网络可达且中位 TTFB 低于 3,000 ms，或出现可恢复失败、限流、服务器异常 |
| `POOR` | 主路径成功率低于 60%、中位 TTFB 至少 3,000 ms，或配置的 API 路径返回 404 |
| `BLOCKED` | 主路径没有 HTTP 响应，或代理先返回 HTTP 407 |
| `SKIPPED` | 必须先提供明确的无凭据检测地址，才能测量该 provider |

检测多个客户端时，各客户端保持独立结论：全部阻断时总体为 `BLOCKED`；阻断
与可达客户端并存时总体为 `POOR`；否则采用最弱客户端的结论。开发适配分始终
使用得分最低的服务链路，因此健康的 Claude 链路不会掩盖异常的 Codex 链路。

已跳过的客户端不会拉低其他已测客户端的分数。所有选中客户端均被跳过时，
总体结论为 `UNAVAILABLE`、分数为 0，命令返回状态码 1。

## 规则 v4 的两个维度

总分明确拆成两个维度：

- **AI 交互：80 分。** 使用得分最低的已选服务链路，衡量可达性、无凭据协议
  TTFB、P95 和抖动。
- **工程传输：20 分。** 使用两个小型下载样本和两个全零上传样本，衡量当前
  代理到 Cloudflare 的传输表现。

这个权重会让持续的传输瓶颈明显影响结论，同时不会让 CDN 路径盖过真实的
Codex 或 Claude 协议链路。代码编辑本身发生在本地；网络传输主要影响发送
上下文、接收响应、克隆仓库和安装依赖。

### AI 交互：80 分

| 项目 | 最高分 | 规则 |
| --- | ---: | --- |
| 可达性 | 30 | 成功率 100% 得 30，60–99% 得 19，大于 0% 得 8，否则 0 |
| TTFB 中位数 | 30 | 低于 800 ms 得 30，低于 1,500 得 26，低于 3,000 得 19，低于 5,000 得 12，否则 5 |
| TTFB P95 | 10 | 低于 2,000 ms 得 10，低于 4,000 得 7，低于 6,000 得 4，否则 0 |
| 抖动 | 10 | 低于 200 ms 得 10，低于 500 得 7，低于 1,000 得 4，否则 0 |

阻断链路不会获得延迟和稳定性分。

### 工程传输：20 分

下行和上行各占 10 分。界面显示的速度是该方向有效样本速度的平均值。

| 方向 | 10 分 | 7 分 | 4 分 | 0 分 |
| --- | ---: | ---: | ---: | ---: |
| 下行 | 至少 10 Mbps | 至少 3 Mbps | 至少 1 Mbps | 低于 1 Mbps |
| 上行 | 至少 5 Mbps | 至少 1 Mbps | 至少 0.3 Mbps | 低于 0.3 Mbps |

对应评级为“传输舒适”“轻度受限”“明显受限”和“严重受限”。这些阈值
只描述限量 Cloudflare 样本，不是通用的运营商网络评级。

跳过或无法取得样本时，该方向使用中性的 10 分，避免防火墙、离线报告或
`--no-bandwidth` 产生虚假扣分；维度会标记为“未测量”，JSON 同时给出可信度。
未完成样本仍可提供估算值，但会降低可信度，而且不能单独认定为已确认瓶颈。

## 结论限幅与标签

原始维度总分同时受到服务证据与传输证据的限幅。服务限幅为：

- 可用的 `FAIR` 链路最高 89 分；
- 临时不可用的 `FAIR` 和 `POOR` 链路最高 64 分；
- `BLOCKED` 链路为 0 分。

传输限幅必须有重复证据：

- 任一方向的两个有效样本均为“明显受限”或更差时，总分最高 79；
- 任一方向的两个有效样本均为“严重受限”时，总分最高 74；
- 只有一个有效的“严重受限”估算值时，总分最高 89，但不视为已确认。

最终采用服务限幅和传输限幅中更低的一个。这样既不会让优秀 TTFB 掩盖持续
不可用的传输路径，也不会凭一个部分样本就下过强结论。

| 分数 | 标签 |
| ---: | --- |
| 90–100 | 舒适 |
| 75–89 | 良好 |
| 65–74 | 可用 |
| 0–64 | 受限 |

运行 `ipcheck --explain-score` 可以查看两个维度、各个项目和全部限幅。
JSON schema 3 会在 `developer_readiness.dimensions` 和
`developer_readiness.score_breakdown` 中提供相同的计算数据。
